## SNS에서 말하는 것처럼 쉽지 않았다

클로드 코드(Claude Code)를 처음 써봤을 때 드는 생각은 단순했다. *"이거 잘 쓰면 내가 원하는 소프트웨어를 직접 만들 수 있겠는데?"*

AI를 이용하면 업무 효율성을 개선하는 도구를 직접 만들 수 있을 것 같았다. SNS에서 보던 영상들도 그렇게 이야기했다. 프롬프트 한 줄 입력하면 그럴듯한 앱이 뚝딱 만들어지는 영상들.

그런데 직접 써보니 달랐다.

물론 간단한 건 된다. 하지만 내가 원하는 수준, 즉 실제로 쓸 수 있는 물건을 만들려고 하면 이야기가 달라졌다. AI는 내 의도를 절반쯤만 이해하고 엉뚱한 방향으로 달려가거나, 앞에서 한 작업을 뒤에서 뒤집어 놓거나, "완료했습니다"라고 말해놓고 실제로는 제대로 동작하지 않는 코드를 내놓기도 했다. 프롬프트를 잘 짜면 좀 나아지긴 했지만, 그것만으로는 한계가 있었다.

그래서 클로드 코드를 좀 더 파고들었다.

---

## 하네스 엔지니어링을 알게 됐다

공부하다 보니 **'하네스 엔지니어링(Harness Engineering)'** 이라는 개념을 만났다.

말(馬)과 마구(馬具)의 관계를 생각하면 이해하기 쉽다.

아무리 능력 좋은 말이 있어도, 마구 없이는 그 힘을 제대로 쓸 수 없다. 마구가 있어야 말의 에너지가 올바른 방향으로 전달되고, 말도 사람도 서로 안전하게 일할 수 있다. 클로드 코드(AI)가 아무리 뛰어나도, 아무런 구조 없이 쓰면 힘이 엉뚱한 방향으로 새거나 사고가 난다. **하네스는 그 마구에 해당한다.** AI가 무엇을 해도 되는지 규칙을 정하고, 작업이 명확한 순서를 따라가게 하며, 결과물이 제대로 만들어졌는지 증거를 남기는 틀이다.

개념은 금방 이해했다. 하지만 실제 하네스가 어떻게 생겼고, 어떤 파일이 어떻게 작동하는지를 이해하는 건 다른 문제였다. 개발자가 아니다 보니 용어도 낯설고, 작동 구조도 처음엔 와닿지 않았다.

그래서 여러 번 직접 만들어봤다.

---

## 직접 만들어봤다

이해하는 가장 빠른 방법은 직접 만들어보는 것이었다. 부수기도 하고, 다시 만들기도 하면서, 결국 기초적인 하네스의 작동 구조를 이해하게 됐다.

그 이해를 바탕으로 만든 것이 **QQ-Harness**다.

거창한 것이 아니다. 나 같은 비개발자가 클로드 코드로 실제 개발 작업을 할 때 쓸 수 있는, 가볍고 실용적인 범용 하네스다. 기획부터 출시까지 9단계 절차를 따라가게 하고, AI가 비밀 정보를 건드리거나 위험한 작업을 하지 못하도록 자동으로 막아주며, "완료했다"는 말 대신 실제 검증 결과를 남기도록 설계했다.

만들면서 가장 신경 쓴 건 하나다: **나처럼 개발 배경이 없는 사람도 이해하고 실제로 쓸 수 있어야 한다는 것.**

---

## QQ-Harness가 하는 일

클로드 코드는 기본 상태에서도 강력하지만, 제어되지 않은 상태에서는 다음과 같은 문제가 생길 수 있다.

- 비밀번호나 API 키가 담긴 파일을 마음대로 읽는다
- "삭제" 같은 위험한 명령을 실행한다
- 증거 없이 "완료"를 선언한다
- 작업 이후 정리하지 않고 잡동사니를 남긴다

QQ-Harness는 이런 문제를 구조적으로 막는다.

- 🔒 **안전 난간** — 비밀 파일 접근과 위험한 명령이 자동으로 차단된다
- 🧭 **단계별 절차** — 명세 → 기획 → 개발 → 검증 → 출시 등 9단계를 순서대로 따라간다
- 🧾 **증거 보존** — "완료"는 통과한 검사 결과가 있을 때만 인정된다
- 🧹 **자동 정돈** — AI가 작업 후 스스로 뒷정리하도록 유도한다

## 어떻게 쓰나요?

클로드 코드가 설치되어 있으면 다른 준비는 필요 없다.

1. [QQ-Harness GitHub 저장소](https://github.com/QriusQuokka/QQ-Harness)에서 ZIP 파일을 받는다 (초록색 *Code → Download ZIP* 버튼)
2. 새 프로젝트 폴더에 압축을 푼다
3. 그 폴더를 클로드 코드로 연다
4. 클로드 코드를 한 번 재시작한다
5. 평소처럼 클로드와 대화를 시작한다

이게 전부다. 하네스는 백그라운드에서 자동으로 작동한다.

설치 이후 첫 확인 방법: 클로드에게 *"`.env` 파일 내용 보여줘"* 라고 해보자. 차단 메시지가 뜨면 하네스가 정상 작동하는 것이다.

### 2026-06-17 — 비개발자용 WBS 대시보드 추가

**무엇이 불편했나.** 하네스는 명세(spec)·실행계획(plan) 같은 문서를 아주 꼼꼼하게 만들어준다. 품질엔 만족했지만, **기획자 입장에서 읽기가 어려웠다.** 전체 작업이 어떻게 나뉘어 있고, 각 파트가 무슨 일을 하며, **지금 무엇을 하고 있는지**를 한눈에 파악하기 힘들었다.

**무엇을 바꿨나.** PM·기획자가 쓰는 **WBS(작업 분해 구조)** 처럼, 비개발자도 이해할 수 있는 언어로 된 설명 자료를 하네스가 **자동으로 만들어주도록** 추가했다. 설계할 때 가장 신경 쓴 원칙은 세 가지다.

- 🧱 **원본은 안 건드린다** — 기존 spec·plan은 그대로 두고, 그것을 *쉬운 말로 번역·요약한 새 자료*를 따로 만든다. 내용이 충돌하면 **항상 원본이 맞다.**
- 🔄 **알아서 최신 상태를 유지한다** — 작업 단계가 진행될 때마다 이 자료가 스스로 갱신된다. 기획자가 아무 때나 열어도 "지금 여기"가 보인다.
- 🗣️ **전문용어를 쓰지 않는다** — "JWT 미들웨어"가 아니라 "로그인 기능"처럼, 결과 중심으로 서술한다. 추측은 넣지 않는다.

**무엇이 가능해졌나.** 기획자는 **`docs/PROJECT_MAP.md` 파일 하나만 열면** 코드나 기술 문서를 보지 않고도 전체 구조 + 진행 상황 + 다음 할 일을 파악할 수 있다. 작업 트리는 ✅완료 · 🔄진행 · ⏳예정 · ⛔막힘으로 표시된다. 더 깊이 보고 싶으면 각 항목에 달린 원본 링크로 내려가면 된다. 실제 프로젝트에서는 `/brief` 를 한 번 실행하면 초안이 채워지고, 이후엔 작업이 진행될 때마다 자동으로 갱신된다.
